### [1] SQL Injection via string interpolation trong `$queryRawUnsafe` — `recommendations.repository.ts:52-86`
**Category**: Security
**Rule**: OWASP A03 — Injection
**Problem**: Hàm `findByUserVector` dùng `$queryRawUnsafe` với `vectorLiteral` được ghép trực tiếp vào SQL string qua template literal. Mặc dù `userVector` đến từ kết quả tính toán nội bộ, đây là pattern cực kỳ nguy hiểm. Nếu bất kỳ upstream input nào (ví dụ embedding từ user-controlled text) ảnh hưởng tới `vectorLiteral`, attacker có thể inject SQL tùy ý.
```ts
// recommendations.repository.ts:52
const vectorLiteral = `[${userVector.join(',')}]`;
// ...
return this.prisma.$queryRawUnsafe<RawMovieRow[]>(
  `... (me.embedding <=> '${vectorLiteral}'::vector) ...`,
);
```
**Risk**: Full SQL injection → đọc/xóa/sửa dữ liệu toàn bộ database. Với pgvector extension, attacker có thể inject `'[1,2,3]'::vector) -- \` để escape khỏi vector context.
**Fix**:
```ts
// before — string interpolation trong SQL
const vectorLiteral = `[${userVector.join(',')}]`;
return this.prisma.$queryRawUnsafe<RawMovieRow[]>(
  `SELECT ... WHERE me.embedding <=> '${vectorLiteral}'::vector ...`,
);

// after — dùng $queryRaw với tagged template (parameterized)
// Prisma không hỗ trợ trực tiếp vector casting nên dùng Prisma.raw cho phần type cast
const vectorParam = `[${userVector.map(Number).join(',')}]`;
return this.prisma.$queryRaw<RawMovieRow[]>(
  Prisma.sql`
    SELECT m.id, m.title, ...
    FROM movie_embeddings me
    JOIN movies m ON m.id = me.movie_id
    WHERE me.movie_id NOT IN (${Prisma.join(excludeIds)})
    ORDER BY me.embedding <=> ${Prisma.raw(`'${vectorParam}'::vector`)}
    LIMIT ${limit}
  `,
);
// Lưu ý: vectorParam phải validate chỉ chứa số và dấu phẩy trước khi dùng Prisma.raw
```

### [2] Tài khoản bị ban (`isActive=false`) vẫn có thể dùng access token — `jwt.strategy.ts:70-74`
**Category**: Security
**Rule**: OWASP A07 — Identification and Authentication Failures
**Problem**: `JwtStrategy.validate()` trả về user payload mà không kiểm tra `user.isActive`. Admin có thể ban user qua `PATCH /admin/users/:id/status` nhưng access token hiện tại (có thể còn sống đến 1h) vẫn hoạt động bình thường.
```ts
// jwt.strategy.ts:70-74
async validate(payload: { sub: string; email: string }) {
  const user = await this.usersService.findByIdWithRoles(payload.sub);
  const roles = user?.roles?.map((ur) => ur.role.name) || [];
  return { userId: payload.sub, email: payload.email, roles };
  // ❌ Không check user?.isActive
}
```
**Risk**: Admin ban user nhưng user vẫn truy cập được tất cả resources trong vòng tối đa 1h. Đặc biệt nguy hiểm trong trường hợp tài khoản bị compromise.
**Fix**:
```ts
// before
async validate(payload: { sub: string; email: string }) {
  const user = await this.usersService.findByIdWithRoles(payload.sub);
  const roles = user?.roles?.map((ur) => ur.role.name) || [];
  return { userId: payload.sub, email: payload.email, roles };
}

// after
async validate(payload: { sub: string; email: string }) {
  const user = await this.usersService.findByIdWithRoles(payload.sub);
  if (!user || !user.isActive) {
    throw new UnauthorizedException('Account is inactive or not found');
  }
  const roles = user.roles?.map((ur) => ur.role.name) || [];
  return { userId: payload.sub, email: payload.email, roles };
}
```

### [3] `watch-history` body bypass hoàn toàn ValidationPipe — `watch-history.controller.ts:34-39`
**Category**: Security / Validation
**Rule**: OWASP A03 — Injection, CWE-20 Improper Input Validation
**Problem**: Controller nhận `@Body() body: unknown` rồi tự `JSON.parse(body)` để "support sendBeacon". Điều này hoàn toàn bypass global `ValidationPipe` — không có whitelist, không có type checking, không có constraint validation nào được thực hiện. Attacker có thể gửi `progressSeconds: -999999` hoặc `movieId: "'; DROP TABLE--"` mà không bị chặn.
```ts
// watch-history.controller.ts:34-39
async saveProgress(@User() user: { userId: string }, @Body() body: unknown) {
  // Handle sendBeacon sending body as text/plain string
  const dto = (
    typeof body === 'string' ? JSON.parse(body) : body
  ) as SaveProgressDto;
  return this.watchHistoryService.saveProgress(user.userId, dto);
}
```
**Risk**: Unvalidated input vào database. `progressSeconds` có thể là NaN/Infinity/âm. `movieId` có thể là bất kỳ string nào. Type cast `as SaveProgressDto` không thực sự validate.
**Fix**:
```ts
// before — bypass validation với JSON.parse thủ công
async saveProgress(@User() user: { userId: string }, @Body() body: unknown) {
  const dto = (typeof body === 'string' ? JSON.parse(body) : body) as SaveProgressDto;
  return this.watchHistoryService.saveProgress(user.userId, dto);
}

// after — dùng custom pipe để handle cả JSON string lẫn object
// Tạo ParseJsonBodyPipe:
@Injectable()
export class ParseJsonBodyPipe implements PipeTransform {
  async transform(value: unknown) {
    const parsed = typeof value === 'string' ? JSON.parse(value) : value;
    return plainToInstance(SaveProgressDto, parsed);
    // Validation sẽ do global ValidationPipe xử lý sau
  }
}

// Controller:
async saveProgress(
  @User() user: { userId: string },
  @Body(new ParseJsonBodyPipe()) dto: SaveProgressDto,
) {
  return this.watchHistoryService.saveProgress(user.userId, dto);
}
```

### [4] `ValidationPipe` thiếu `forbidNonWhitelisted: true` — `main.ts:27-32`
**Category**: Validation
**Problem**: Global `ValidationPipe` chỉ có `whitelist: true` (strip unknown fields) nhưng không có `forbidNonWhitelisted: true`. Điều này có nghĩa là client gửi các field không hợp lệ sẽ bị silently strip thay vì reject — khó debug và có thể che giấu các attack attempt.
```ts
// main.ts:27-32
app.useGlobalPipes(
  new ValidationPipe({
    whitelist: true,
    transform: true,
    // ❌ forbidNonWhitelisted: true bị thiếu
  }),
);
```
**Risk**: Client có thể gửi malformed requests mà không nhận được feedback. Các field typo trong request body sẽ bị im lặng bỏ qua, gây khó debug.
**Fix**:
```ts
// before
new ValidationPipe({ whitelist: true, transform: true })

// after
new ValidationPipe({
  whitelist: true,
  forbidNonWhitelisted: true,
  transform: true,
  transformOptions: { enableImplicitConversion: true },
})
```

### [5] Auth routes thiếu rate limiting chuyên biệt — `auth.controller.ts`, `app.module.ts`
**Category**: Security
**Problem**: Global `ThrottlerModule` cấu hình ở mức `100 req/60s` (1.67 req/s) — quá lỏng cho auth routes. `/auth/login` và `/auth/register` cần rate limit nghiêm hơn (ví dụ 5-10 req/60s per IP) để ngăn brute-force và credential stuffing. Hiện tại không có `@Throttle()` decorator nào trên auth controller.
```ts
// app.module.ts:56-61
ThrottlerModule.forRoot([{
  ttl: 60000,
  limit: 100,  // ❌ 100 req/min cho tất cả routes kể cả login
}]),
```
**Risk**: Brute-force tấn công mật khẩu. Với 100 req/min, attacker có thể thử ~6000 passwords/giờ.
**Fix**:
```ts
// app.module.ts — thêm profile riêng cho auth
ThrottlerModule.forRoot([
  {
    name: 'default',
    ttl: 60000,
    limit: 100,
  },
  {
    name: 'auth',
    ttl: 60000,
    limit: 5,
  },
]),

// auth.controller.ts — áp dụng throttle chặt hơn
@Post('login')
@Throttle({ auth: { limit: 5, ttl: 60000 } })
async login(...) {}

@Post('register')
@Throttle({ auth: { limit: 3, ttl: 60000 } })
async register(...) {}
```

### [6] Race condition trong `watch-history.saveProgress` — `watch-history.service.ts:18-46`
**Category**: Database
**Problem**: Pattern findFirst → update/create có race condition: hai requests đồng thời có thể đều thấy `existing = null` và cùng `create` → unique constraint violation. Nên dùng `upsert` với composite key `@@unique([userId, movieId, episodeId])` đã có trong schema.
```ts
// watch-history.service.ts:18-46
const existing = await this.prisma.watchHistory.findFirst({ where: {...} });
if (existing) {
  result = await this.prisma.watchHistory.update(...);  // race condition
} else {
  result = await this.prisma.watchHistory.create(...);  // có thể duplicate
}
```
**Risk**: Unhandled `P2002` unique constraint violation trong production → 500 error cho user đang xem phim.
**Fix**:
```ts
// before — findFirst + create/update pattern
const existing = await this.prisma.watchHistory.findFirst({ ... });
let result;
if (existing) { result = await this.prisma.watchHistory.update(...); }
else { result = await this.prisma.watchHistory.create(...); }

// after — atomic upsert
result = await this.prisma.watchHistory.upsert({
  where: {
    userId_movieId_episodeId: {
      userId,
      movieId: dto.movieId,
      episodeId: dto.episodeId ?? null,
    },
  },
  create: {
    userId,
    movieId: dto.movieId,
    episodeId: dto.episodeId ?? null,
    progressSeconds: dto.progressSeconds,
    isFinished: dto.isFinished,
  },
  update: {
    progressSeconds: dto.progressSeconds,
    isFinished: dto.isFinished,
    lastWatchedAt: new Date(),
  },
});
```

### [9] `getTrendingMovieIds` trong repository dùng `$queryRaw` với INTERVAL injection — `recommendations.repository.ts:228-236`
**Category**: Database / Security
**Problem**: Query dùng `INTERVAL '${days} days'` với `days` là parameter số. Mặc dù `days` là `number` type, khi dùng với `$queryRaw` tagged template, Prisma parameterize số nhưng INTERVAL interpolation có thể không được xử lý đúng cách trong tất cả versions.
```ts
// recommendations.repository.ts:228-236
const rows = await this.prisma.$queryRaw<{ movie_id: string }[]>`
  SELECT movie_id, COUNT(*) AS watch_count
  FROM watch_history
  WHERE last_watched_at >= NOW() - INTERVAL '${days} days'
  GROUP BY movie_id
  ORDER BY watch_count DESC
  LIMIT ${limit}
`;
```
**Risk**: Prisma `$queryRaw` với tagged template sẽ attempt to parameterize `days` nhưng INTERVAL syntax `'${param} days'` không hợp lệ với parameterized queries trong PostgreSQL. Query có thể fail silently hoặc produce unexpected results.
**Fix**:
```ts
// after — dùng date arithmetic thay vì INTERVAL string interpolation
const since = new Date(Date.now() - days * 24 * 60 * 60 * 1000);
const rows = await this.prisma.$queryRaw<{ movie_id: string }[]>`
  SELECT movie_id, COUNT(*) AS watch_count
  FROM watch_history
  WHERE last_watched_at >= ${since}
  GROUP BY movie_id
  ORDER BY watch_count DESC
  LIMIT ${limit}
`;
```

## Positive Findings (điểm làm tốt)

1. **JWT RS256 + per-session key pair** — Approach asymmetric với fresh RSA-2048 key pair mỗi login session, public key lưu Redis với TTL khớp access token — đây là implementation bảo mật cao hơn standard JWT nhiều.
2. **Refresh token rotation + token reuse detection** — Khi phát hiện token reuse (`bcrypt.compare` fail), service revoke toàn bộ session ngay lập tức (line 171 auth.service.ts). Pattern này chống token theft rất tốt.
3. **Stripe webhook idempotency** — `handleWebhook` kiểm tra Redis trước khi process event (`IDEMPOTENCY_PREFIX + event.id`) — đảm bảo webhook được process đúng 1 lần, ngăn duplicate subscriptions.
4. **RabbitMQ DLQ với retry logic** — Exponential backoff (5s → 15s → 60s), max 3 retries, messages fail vĩnh viễn đi vào DLQ để audit — production-ready pattern.
5. **Global exception filters** — `HttpExceptionFilter` bắt tất cả exceptions, `PrismaExceptionFilter` translate P2002/P2025/P2003 sang HTTP codes phù hợp. Kết hợp hai filter coverage tốt.
6. **Env validation** — `EnvironmentVariables` class với `validateSync` tại startup. App sẽ fail-fast nếu required env vars missing.
7. **Repository pattern nhất quán** — Tất cả modules lớn (auth, movies, admin, subscriptions) đều tách biệt Repository/Service/Controller rõ ràng.
8. **`$executeRaw` / `$queryRaw` tagged templates cho safe queries** — `embedding.service.ts` và `recommendations.repository.ts` (các collaborative queries) đúng sử dụng Prisma tagged template literals để parameterize.
9. **Redis caching với TTL hợp lý** — Recommendations có TTL hierarchy (userProfile 1h, forYou 1h, similar 6h, trending 30m), admin stats 5-10 phút — hợp lý với use case.
10. **Test coverage đủ tốt cho core modules** — `auth.service.spec.ts`, `movies.service.spec.ts`, `watch-history.service.spec.ts`, `watchlist.service.spec.ts` đều có `describe` grouping, `beforeEach` với `clearAllMocks`, và test cả happy path lẫn error cases.
